# Docker for ML — Mental Model, Concepts & Reference

---

## Part 1 — Why Docker? (The Mental Model)

### The "works on my machine" problem

Every ML engineer has lived this:

```
Data scientist: "The model works perfectly, here's the code."
Engineer:       "It crashes on the server."
Data scientist: "But it works on my laptop!"

Laptop:  Python 3.10, scikit-learn 1.2.0, numpy 1.24, CUDA 11.8
Server:  Python 3.8,  scikit-learn 1.0.2, numpy 1.21, no CUDA
```

The model isn't broken. The environment is different. The code assumes a specific
Python version, specific library versions, specific system libraries. When the
environment changes, the behavior changes — or the code crashes entirely.

**Docker solves this by packaging the environment with the code.**

Instead of shipping code and hoping the environment matches, you ship a container:
a self-contained unit that includes the OS libraries, Python version, all packages,
and your code. It runs identically everywhere Docker is installed — your laptop,
a colleague's machine, a cloud server.

---

### What a container actually is

A common mental model mistake: thinking containers are mini virtual machines.
They're not. They're isolated processes.

```
Virtual Machine                    Container
──────────────                     ──────────────
Full OS (2-4 GB)                   Your app + dependencies (~500 MB)
Boots in minutes                   Starts in seconds
Strong isolation (separate kernel) Process-level isolation (shared kernel)
High overhead                      Low overhead
```

A container is a process that:
- Has its own filesystem (isolated from the host)
- Has its own network namespace
- Has its own process namespace
- But shares the host OS kernel

When you run a Python container, you're running a Python process in an isolated
environment — not booting a new operating system.

---

### Images vs containers

This distinction is important:

**Image:** A read-only template. A snapshot of an environment at a point in time.
Built once, used many times. Stored in a registry (Docker Hub, etc.).

```dockerfile
# This is a Dockerfile — instructions for building an image
FROM python:3.12-slim        # start from official Python 3.12 image
RUN pip install fastapi      # add FastAPI
COPY api/ ./api/             # add your code
CMD ["uvicorn", "main:app"]  # set the default command
```

Running `docker build` creates an image from these instructions.

**Container:** A running instance of an image. You can run many containers from the
same image simultaneously. Containers are ephemeral — they start, do work, stop.
Any data written inside a container disappears when the container stops (unless you use volumes).

```
Image (blueprint)    →   Container (instance running)
                     →   Container (another instance)
                     →   Container (yet another instance)
```

---

### The ML-specific challenge: training vs serving

In a typical web app, the same team builds and deploys the service. In ML, training
and serving are separate activities with different requirements:

```
Training (pipeline)             Serving (API)
───────────────────             ─────────────
Runs occasionally               Runs continuously
Needs Prefect, big compute      Needs only inference libs
Developer runs it locally       Deployed on a server
Output: model in registry       Input: model from registry
```

**This project reflects that separation:**

- **Pipeline runs locally** — developers train with Prefect for visibility,
  iterate quickly, use the full dev environment. Containerizing Prefect's
  ephemeral server adds complexity with no benefit at this scale.

- **API runs in Docker** — the serving layer is containerized for reproducibility.
  The same container that works on your laptop works on any server.

The two components share state through the **MLflow registry** — a SQLite file
on the local filesystem that the API reads via a bind mount.

---

### Volumes and bind mounts — sharing state with containers

A container has its own isolated filesystem. To share files between the host and
a container, you use **mounts**:

**Named volume** — Docker manages the storage internally:
```yaml
volumes:
  mlflow_store:           # Docker creates and manages this

services:
  api:
    volumes:
      - mlflow_store:/app/data   # mounted inside container
```
Use when: data is generated by another container (e.g., a database container).

**Bind mount** — maps a specific host directory into the container:
```yaml
services:
  api:
    volumes:
      - ./pipeline:/app/pipeline   # host path → container path
```
Use when: the host already has the data and you want the container to read it.

**This project uses a bind mount** because the pipeline runs locally and writes
the MLflow registry to `./pipeline/`. The API container reads from the same files:

```
Host:                              API container:
./pipeline/                   →    /app/pipeline/
  mlflow_trip_duration.db              mlflow_trip_duration.db
  mlruns/                              mlruns/
    1/models/m-xxx/artifacts/            1/models/m-xxx/artifacts/
```

---

### The artifact path problem — a real production lesson

Using a local SQLite MLflow registry in Docker reveals a real-world problem:
**artifact paths are stored as absolute host paths** in the database.

When the pipeline runs locally, MLflow stores:
```sql
-- In mlflow_trip_duration.db
model_versions: storage_location = '/home/silva/.../pipeline/mlruns/1/models/m-xxx/artifacts'
runs: artifact_uri = '/home/silva/.../pipeline/mlruns/1/run_id/artifacts'
```

Inside the Docker container, those paths don't exist. The bind mount is at `/app/pipeline/`,
so the correct container path is:
```
/app/pipeline/mlruns/1/models/m-xxx/artifacts
```

**The fix:** `model_loader.py` reads the absolute path from SQLite directly and remaps it
to the container path before loading:

```python
MLFLOW_ARTIFACTS_ROOT = os.getenv("MLFLOW_ARTIFACTS_ROOT")

def _remap(path: str) -> str:
    """Replace host prefix with container pipeline path."""
    if not MLFLOW_ARTIFACTS_ROOT or not path:
        return path
    idx = path.find("/mlruns/")
    return MLFLOW_ARTIFACTS_ROOT + path[idx:] if idx >= 0 else path
```

Set in docker-compose:
```yaml
environment:
  - MLFLOW_ARTIFACTS_ROOT=/app/pipeline
```

**The production lesson:** SQLite as an MLflow backend is fine for local development
but creates coupling between the host filesystem and any container that reads it.
In production, you'd use a proper MLflow tracking server with object storage
(S3, GCS) — then artifact paths are URLs, not local paths, and they work everywhere.

## Part 2 — Docker Concepts

### The Dockerfile — layer by layer

```dockerfile
FROM python:3.12-slim
```

Every image starts from a base. `python:3.12-slim` is an official image from Docker Hub —
Debian Linux with Python 3.12 pre-installed, stripped of unnecessary packages.
`slim` is ~150MB. The full `python:3.12` is ~1GB. For ML, use `slim` and add only
what you need.

```dockerfile
RUN pip install uv
```

`RUN` executes a command during the build and saves the result as a new layer.
Each `RUN` creates a layer. Layers are cached — if this line hasn't changed,
Docker reuses the cached layer instead of re-running `pip install uv`.

**Layer caching is why Dockerfile order matters:**
```dockerfile
# WRONG — cache-busting order
COPY . .                             # copies all source files (changes often)
RUN pip install -r requirements.txt  # re-runs every time any source changes

# CORRECT — cache-friendly order
COPY requirements.txt .              # only copy requirements first (changes rarely)
RUN pip install -r requirements.txt  # only re-runs when requirements change
COPY . .                             # copy source code last
```

In this project (`api/Dockerfile`):
```dockerfile
FROM python:3.12-slim

RUN pip install uv               # Layer 1: uv installed (cached after first build)

WORKDIR /app

COPY shared/ ./shared/           # Layer 2: shared code (changes occasionally)
COPY api/ ./api/                 # Layer 3: api code (changes often)

WORKDIR /app/api

RUN uv pip install --system -r requirements.txt  # Layer 4: deps (cached when unchanged)

CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]
```

---

### The build context — what Docker can see

When you run `docker build`, Docker sends a **build context** to the Docker daemon —
all the files it's allowed to use in `COPY` instructions.

```yaml
# docker-compose.yml
services:
  api:
    build:
      context: .        # build context = 4-Deploy-Online/ directory
      dockerfile: api/Dockerfile
```

With `context: .` (the `4-Deploy-Online/` directory), the Dockerfile can `COPY`:
- `shared/` ✅
- `api/` ✅
- `pipeline/` ✅
- Files outside `4-Deploy-Online/` ❌

This is why `shared/` is at the `4-Deploy-Online/` level — the `api/Dockerfile`
needs to copy it, so it must be in the build context.

**`.dockerignore`** excludes files from the build context:
```
**/__pycache__
**/*.pyc
**/.venv
**/mlruns            ← don't bake model artifacts into image (they come via bind mount)
**/mlflow_*.db       ← don't bake MLflow DB into image
```

Excluding `mlruns/` is important — model artifacts are hundreds of MB and change
every time you retrain. They belong on the filesystem (accessed via bind mount),
not baked into the image.

---

### docker-compose — single-service orchestration

Docker Compose describes the system in `docker-compose.yml` and manages containers
with simple commands. In this project, only the API runs in Docker:

```yaml
services:

  api:
    build:
      context: .
      dockerfile: api/Dockerfile
    volumes:
      - ./pipeline:/app/pipeline   # bind mount: local registry → container
    environment:
      - MLFLOW_ARTIFACTS_ROOT=/app/pipeline   # remap host artifact paths
    ports:
      - "8000:8000"               # host_port:container_port
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
      interval: 30s
      start_period: 60s           # MLflow migrations take time on first start
```

---

### Environment variables — configuring containers

Containers should be configurable without rebuilding the image.
Environment variables are the standard mechanism:

```python
# api/model_loader.py
MLFLOW_ARTIFACTS_ROOT = os.getenv("MLFLOW_ARTIFACTS_ROOT")
# None locally → use absolute paths from MLflow DB directly
# "/app/pipeline" in Docker → remap host paths to container paths
```

Same code. Different behavior. No code change needed between local and Docker.

This is also how you'd point the system at a remote MLflow server:
```bash
export MLFLOW_TRACKING_URI=http://your-mlflow-server:5000
docker compose up api
```

### The API Dockerfile

Only the API is containerized. The pipeline runs locally (Prefect requires a
dedicated server process to run in Docker — local execution is the right pattern
for training, which is a development-time activity).

```dockerfile
FROM python:3.12-slim

RUN pip install uv

WORKDIR /app

COPY shared/ ./shared/    # feature engineering — shared with local pipeline
COPY api/ ./api/

WORKDIR /app/api

RUN uv pip install --system -r requirements.txt
```

The `pipeline/Dockerfile` exists (for documentation and future CI use) but is
not in `docker-compose.yml`.

### The artifact path remapping — full code

```python
# api/model_loader.py

_ARTIFACTS_ROOT = os.getenv("MLFLOW_ARTIFACTS_ROOT")  # set in docker-compose

def _remap(path: str) -> str:
    if not _ARTIFACTS_ROOT or not path:
        return path
    idx = path.find("/mlruns/")
    return _ARTIFACTS_ROOT + path[idx:] if idx >= 0 else path

def _get_storage_location(version: str) -> str:
    # MLflow 3.x stores storage_location in SQLite — read it directly
    db_path = MLFLOW_TRACKING_URI.replace("sqlite:///", "")
    conn = sqlite3.connect(db_path)
    row = conn.execute(
        "SELECT storage_location FROM model_versions WHERE name=? AND version=?",
        (MODEL_NAME, version)
    ).fetchone()
    conn.close()
    return row[0] if row else None

# At load time:
if _ARTIFACTS_ROOT:
    storage_loc = _get_storage_location(mv.version)
    model_path = _remap(storage_loc)           # /home/silva/.../mlruns → /app/pipeline/mlruns
    model = mlflow.sklearn.load_model(model_path)
else:
    model = mlflow.sklearn.load_model(f"models:/{MODEL_NAME}@{MODEL_ALIAS}")
```

### The complete workflow

```bash
# Step 1: train locally (pipeline uses Prefect + MLflow)
cd 4-Deploy-Online/pipeline
python main.py --sample-size 500000 --tune --promote
# → writes mlflow_trip_duration.db and mlruns/ to ./pipeline/

# Step 2: build the API image
cd 4-Deploy-Online
docker compose build api

# Step 3: start the API
docker compose up api
# → bind-mounts ./pipeline into /app/pipeline
# → reads @champion from the MLflow registry
# → remaps host artifact paths to /app/pipeline/...

# Step 4: test
curl -X POST http://localhost:8000/predict \
  -H "Content-Type: application/json" \

       "RatecodeID": 1, "trip_distance": 2.5, "payment_type": 1}'

# Step 5: retrain when needed
cd pipeline && python main.py --sample-size 500000 --tune --promote

# Step 6: restart API to pick up new champion
cd .. && docker compose restart api
```

## Part 4 — The Bigger Picture

### What you gain from Docker in ML

| Without Docker | With Docker |
|----------------|-------------|
| "It works on my machine" | Runs identically everywhere |
| Manual dependency management | `docker build` reproduces the exact environment |
| Implicit Python version | `FROM python:3.12-slim` is explicit |
| Shared global packages conflict | Isolated container per service |
| "What version of sklearn is serving?" | Image SHA tells you exactly |

### What this setup doesn't solve (and what production does)

This setup uses SQLite + bind mounts. It works for a single developer.
In production you'd need:

| Problem | Production solution |
|---------|-------------------|
| SQLite paths are host-absolute | MLflow tracking server + S3/GCS artifacts |
| Bind mount is machine-specific | Container registry + remote artifact store |
| Single machine | Kubernetes / ECS for scaling |
| Manual retrain trigger | CI/CD pipeline on schedule |

The Docker concepts here — images, containers, volumes, bind mounts, compose,
environment variables — are identical in production. Only the infrastructure changes.

## Quick Reference

```bash
# Build
docker compose build api                      # build API image

# Run
docker compose up api                         # start API (foreground)
docker compose up -d api                      # start API (background)

# Inspect
docker compose ps                             # running services
docker compose logs api                       # API logs
docker compose logs -f api                    # follow logs

# Stop
docker compose down                           # stop and remove containers

# Shell into running container
docker compose exec api bash

# Rebuild after code changes
docker compose build api && docker compose restart api

# Check MLflow version parity (local must match Docker)
python -c "import mlflow; print(mlflow.__version__)"
docker run --rm 4-deploy-online-api python -c "import mlflow; print(mlflow.__version__)"
```
